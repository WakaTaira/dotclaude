# Pull Request の作成・更新（日本語）

PR説明文はレビュアーのための読み方ガイドであり、変更の恒久的な記録である。diffを開く前の最初のひと画面で「何が・なぜ変わり・どこから読めばよいか」が伝わることを目指す。

この方針は日本語OSSコミュニティのマージ済みPR約150件（misskey-dev/misskey・VOICEVOX/voicevox・sakura-editor/sakura・rurema/doctree・codeforjapan/decidim-cfj 等）の実地調査に基づく。英語版スキルと共通する原則（サイズゲート・diff反復禁止・AI臭の回避）はここでも適用する。

## 日本語コミュニティ特有の絶対原則

### 1. テンプレート至上主義

日本語OSSの大半は `.github/PULL_REQUEST_TEMPLATE.md` を持ち、**テンプレートを忠実に埋めることが自由形式の名文より優先される**。

- 作成前に必ず `CONTRIBUTING.md` とPRテンプレートを読む。テンプレートがあればこのスキルの雛形は使わず、テンプレートの構成に従う。
- `<!-- 必須 -->` 等の指示コメントは指示として厳守する。テンプレート内のHTMLコメントは、リポジトリの慣習（直近のマージ済みPRを確認）に合わせて残すか消すか判断する。迷ったら残す。
- 該当なしのセクションは削除せず「（なし）」「なし」と明記する文化がある（VOICEVOX等）。テンプレートの指示（「なければ省略可」等）があればそちらに従う。
- チェックリストは実際に確認した項目だけチェックする。未実施項目を偽ってチェックしない。

### 2. 文体：本文はです・ます調、タイトルは常体

調査では日本語PRの約76%が本文にです・ます調を使用。

- **本文**：です・ます調が基本。「〜を追加します」「〜するようにします」「〜を確認済みです」。箇条書き内は体言止め・常体の混在も自然（「既存の慣例に準拠」「対応不要でした」）。
- **タイトル**：常体・体言止め。「〜を追加」「〜を修正」「〜する」「〜対応」。です・ます調のタイトルは書かない。
- 過度な謙譲・冗長な挨拶は不要。「お忙しいところ恐れ入りますが」等は書かない。一方で、判断を仰ぐ箇所は「必要ならご指示ください」のように簡潔に添えるのが好印象。

### 3. タイトル規約はリポジトリに合わせる

調査では prefix なしが約半数、モダンなリポジトリは Conventional Commits（`fix:` `feat:` `chore:` `refactor:`）を採用。

- `git log --oneline -20` と直近のマージ済みPR数件でリポジトリの流儀を確認し、それに合わせる。
- Conventional Commits採用リポジトリでは、prefixは英語・本文説明は日本語のハイブリッドが一般的：`feat: エンジン分離版のビルド経路を追加`、`refactor(frontend): メディアメニューのスタイルを統一`。
- タイトルだけで変更範囲と種別が伝わること。「〜の修正」だけの曖昧タイトルは避け、対象を含める：「チャンク境界計算のoff-by-oneを修正」。

### 4. Issue参照は「関連 Issue」セクション

- 「関連 Issue」（または「関連 issue, PR」）セクションを設けて `- ref: #1234` / `- fixes #1234` / `- resolves #1234` と書くのが最頻出パターン。
- 自動クローズしたい場合のみ `fixes` / `resolves` / `close`。参照だけなら `ref:`。
- 関連する上流（他リポジトリ）のissueは `owner/repo#123` 形式でリンクする。

## 頻出セクション語彙（テンプレートが無い場合の雛形）

調査での頻出見出し：`内容` / `概要` / `背景` / `関連 Issue` / `仕様・動作説明` / `影響範囲` / `テスト内容` / `スクリーンショット・動画など` / `その他`。バイリンガル運用リポジトリでは `What` / `Why` / `Additional info`。

テンプレートが無いリポジトリでは以下を雛形とし、**英語版スキルのサイズゲートを適用して不要セクションを削る**：

```
## 概要

[具体的な数値・エラー・実例を軸に2文で。1文目に問題、2文目にこのPRの対応。]

## 背景

[なぜこの変更が必要か。エラーメッセージ・誤った出力・欠けている機能を示す。
概要で十分なら省略。上流バグの暫定回避なら、上流issueへのリンクと
「上流の修正が入り次第削除できます」を必ず明記。]

## 変更内容

[設計判断を上から順に。ファイル単位の逐語ナレーションはしない。]

## 影響範囲

[既存の挙動への影響。互換性・移行の要否。破壊的変更は目立つように。]

## テスト内容

[何をどう確認したか具体的に：「typecheck / test / lint 通過」
「Linux実機でバグ再現→パッチ適用で解消を確認」等。
レビュアーが再現できる手順があれば記載。]

## 関連 Issue

- ref: #1234

## その他

[補足・参考資料・スクリーンショット。なければ「（なし）」または省略。]
```

小規模PR（diff 50行未満・単一関心事）なら「概要」と「関連 Issue」だけでよい。説明がdiffより長いのは書きすぎ。

## 書き方の質（日本語での適用）

- **具体性がすべて**：「パフォーマンスを改善」ではなく「起動が2.1秒から0.3秒に短縮」。「一時ファイルが溜まる問題」ではなく「3週間で81,653ファイル・約190GB」。
- **diffから読み取れることは書かない**：説明文はdiffの補集合。「`foo.py`でXを変更し、`bar.py`でYを変更」式のファイル巡りは削る。
- **AI臭のクラスタリングを避ける**：「このPRでは〜」「本PRは〜」で文を始めない。問題・対象コンポーネント・動作から始める。全箇条書きが同一構造になる規則性も避ける。
- **判断保留・見送りの明示**は日本語コミュニティで特に好まれる：「eval.mdにも同様の箇所がありますが、重複ヒットを避けるため今回は見送りました（必要ならご指示ください）」のように、やらなかったことと理由を書く。
- **6ヶ月テスト**：半年後に `git log` からこのPRに辿り着いた他人が「なぜ」を理解できるか。

## 外部コントリビューション時の追加手順

1. `CONTRIBUTING.md`・PRテンプレート・直近のマージ済みPRを3〜5件読み、語彙・文体・テンプレの埋め方を模倣する。
2. 同じ問題のissue/PRを閉じたものも含めて検索し、リンクする。「上流の問題」としてクローズ済みのissueがある場合、議論の蒸し返しではなく暫定緩和策として位置づける。
3. 変更は最小限・削除しやすく。挙動変更には安価ならオプトアウト手段（環境変数等）を付ける。
4. 検証は具体的に：どのチェックが通り、どのプラットフォームで何をE2E確認したか。
5. AI利用の帰属表記（Co-Authored-By等）はリポジトリの文化と自分の方針に従う。日本語OSSでは「Generated with Claude Code」表記のままマージされる例も普通にある。

## 提出前チェックリスト

- [ ] リポジトリのテンプレート・CONTRIBUTINGに従っている
- [ ] タイトルは常体・対象を含む・リポジトリのprefix慣習に一致
- [ ] 本文はです・ます調で統一（箇条書き内の体言止めは可）
- [ ] 概要に具体的な数値・実例がある
- [ ] diffから読み取れる文が無い
- [ ] 「関連 Issue」で参照が張られている（自動クローズの要否も正しい）
- [ ] 該当なしセクションの扱い（「（なし）」明記 or 省略）がリポジトリ慣習と一致
- [ ] チェックリストは実施済み項目のみチェック
- [ ] 暫定対応なら上流リンクと削除条件を明記
- [ ] 6ヶ月テストに合格

## 実行手順

1. 作成か更新かを判定：`gh pr view --json number,title,body,baseRefName,url 2>/dev/null`
2. コンテキスト収集：
   ```
   BASE=$(gh pr view --json baseRefName -q '.baseRefName' 2>/dev/null || echo "main")
   git diff $BASE...HEAD --stat
   git diff $BASE...HEAD
   git log $BASE..HEAD --oneline
   ```
   statだけでなくdiff本体を読む。
3. テンプレート確認：`.github/PULL_REQUEST_TEMPLATE.md`（および `.github/PULL_REQUEST_TEMPLATE/` 配下）を読む。
4. サイズゲートで書くセクションを先に決め、概要（TL;DR）から書く。
5. 提出前チェックリストを1文ずつ適用する。
6. 本文は一時ファイル経由で渡す（インライン `--body` やHEREDOCは使わない）：
   ```
   gh pr create --draft --title "..." --body-file /tmp/pr-body.md
   gh pr edit <number> --title "..." --body-file /tmp/pr-body.md
   ```

既存PRの更新時は変更履歴を書き足すのではなく、base に対する**ブランチの現在の姿**を新規作成のつもりで書き直す（「〜も追加しました」が出たら書き方を誤っている）。
