# relay

設計が固まったあとの実装フェーズを自走させるためのスキルである。

このスキルが発動した時点で、**設計や方針はメインセッションですでに合意済みである**ことを前提とする。発動前にどこまで設計を詰めたかはユーザーの裁量であり、このスキルはその合意内容を所与として実装に着手する。設計自体をここでやり直してはならない。

## 役割分担

このスキルの核心は、レイヤーの分離にある。**relay のコアはメインセッション（fable）である。**

- **メインセッション（fable）= 設計・統括・監査レイヤー**
  全体設計の保持、タスク分解、サブエージェントへの指示、返ってきた成果物の監査を担う。原則としてメインセッションは実装の細部に直接手を下さない。grep 結果の羅列や試行錯誤のログでメインのコンテキストを汚さないことが、監査精度を保つうえで重要である。

- **サブエージェント（relay-*）= 実装・調査レイヤー**
  個別の作業を、それぞれ独立したコンテキストウィンドウで遂行する。

## 発動時のガード

relay-investigator / relay-reviewer は `model: inherit` であり、メインセッションのモデルをそのまま使う。メインが fable / opus 系以外（sonnet、haiku など）で発動された場合、調査・レビューの品質が**警告なく**劣化する。発動時にメインのモデルを確認し、fable / opus 系でなければ作業を始めず、ユーザーに警告して指示を仰ぐこと。

## 実行手順

以下の 5 ステップを順に実行する。ステップを飛ばさないこと。

### Step 1: タスク分解表を作る

サブエージェントを 1 体でもスポーンする前に、必ずタスク分解表を作成し、メッセージとして出力する。フォーマットは以下に固定する。

| # | タスク | 依存 | エージェント | 完了条件 |
|---|---|---|---|---|
| 1 | （何をするか） | なし | relay-investigator | （機械的に判定できる条件） |
| 2 | （何をするか） | #1 | relay-implementer-std | （同上） |

- **依存**: 先行タスクの番号。依存のないタスク同士は並列実行の候補になる
- **エージェント**: 後述の振り分け表の relay-* 7 種のいずれか
- **完了条件**: 「テスト X が通る」「ファイル Y に関数 Z が存在する」など、監査時に機械的に検証できる形で書く。「〜が改善される」のような曖昧な条件は不可

この表が実行フェーズ全体の台帳になる。タスクの追加・分割が必要になったら表を更新してから動く。

### Step 2: エージェントを選び、プロンプトを作る

サブエージェントはゼロから設計しない。`~/.claude/agents/` に定義済みの relay-* から、次の振り分け表に従って選ぶ。モデル・effort・ツール権限・報告フォーマットは各エージェント定義に固定されており、ハーネスが機械的に強制する。

#### 振り分け表

| エージェント | model / effort | 用途 |
|---|---|---|
| relay-investigator | inherit（=メイン）/ high | コードベース・仕様・原因の調査。選択肢と推奨を返す（読み取り専用） |
| relay-reviewer | inherit（=メイン）/ high | 変更差分の品質・整合性・セキュリティ評価（読み取り専用） |
| relay-implementer | opus（4.8）/ high | 複数ファイルにまたがる変更、並行処理・境界条件の実装、複雑なデバッグ |
| relay-implementer-std | sonnet / medium | 既存パターンに沿ったコード生成、リファクタリング |
| relay-codex | sonnet / medium（実装本体は CLIProxyAPI 経由の GPT ヘッドレス Claude Code） | 実装タスクの既定レーン。確定済みの仕様をヘッドレス Claude Code（GPT）に流し、結果を独立検証して返す |
| relay-verifier | sonnet / medium | ビルド・テスト・lint の実行と結果の一次評価（修正はしない） |
| relay-mechanic | haiku / low | ファイル探索、単純置換、定型コードの生成（判断を含まない作業のみ） |

言語レベルの規約・スタイルの問題は、レビューエージェントではなく lint / formatter で機械的に解決する（実行は relay-verifier の検証に含める）。ドメイン特化の知見が必要な場合は既存の汎用エージェント（docs-lookup、architect など）を補助的に併用してよいが、実装フェーズの主経路は relay-* とし、モデル選択の一貫性を崩さないこと。

#### クロスベンダー実装レーン（relay-codex）

relay-codex は実装タスクの既定レーンである。プロンプト必須 4 項目が書き切れ、完了条件に実行可能な検証コマンドが含まれる実装タスクは、原則 relay-codex に流す。実装本体を GPT（CLIProxyAPI 経由のヘッドレス Claude Code）が担うことで、単一ベンダーのモデル群が揃って見逃す問題を拾う。振り分けの判断軸はタスクの大きさではなく裁量の有無である——複数ファイルにまたがっていても仕様が確定していれば relay-codex へ、裁量・設計判断・複雑なデバッグが残っているタスクは relay-implementer 系に割り当てる。relay-codex が「失敗」（プロキシ停止・GPT 利用不可。クォータ枯渇・レート制限を含む）を返した場合は、同じ仕様のまま relay-implementer または relay-implementer-std に振り直す。

#### レビューの 2 軸分離

仕様が文書化されているタスク（完了条件が明確、または grill-me の成果物・PRD がある場合）のレビューは、**仕様軸**と**規約軸**の 2 タスクに分け、relay-reviewer を並列で 2 体スポーンする。軸はスポーン時のプロンプトで指定し、軸の定義とスメルベースラインは relay-reviewer 側に定義済みである。監査では 2 つの報告を軸別のまま並記して扱う。規約は守れているが仕様と違うコードと、仕様どおりだが規約に反するコードが、別々に見えている状態を保つ——軸をまたいだ統合ランキングは、一方の指摘がもう一方を覆い隠す原因になるため行わない。

軽微な変更のレビューは、軸指定なしの relay-reviewer 1 体（総合レビュー）で足りる。

#### プロンプト必須項目

サブエージェントはメインの会話を見られない。スポーン時のプロンプトには、必ず以下の 4 項目をすべて明記する。1 項目でも欠けたままスポーンしてはならない。

1. **目的**: このタスクが全体のどの部分を担うか（1〜2 文）
2. **対象**: 触ってよいファイル・ディレクトリの明示的なリスト。対象外への変更は禁止と書く
3. **制約**: 合意済み設計のうちこのタスクに関わる決定事項、コーディング規約、やってはいけないこと
4. **完了条件**: タスク分解表の完了条件をそのまま転記

#### 構造化報告

報告フォーマットは各エージェント定義に焼き込み済みのため、プロンプトへの転記は不要である。すべての relay-* は次の構造で報告を返すよう定義されており、監査はこの構造との突き合わせで行う。

```
## 結果: 完了 / 部分完了 / 失敗
## 変更ファイル: （フルパスの一覧。調査・検証・レビューでは「なし」）
## 実行コマンドと結果: （検証に使ったコマンドと、その出力の要点）
## 判断したこと: （指示になかったが現場判断した点。なければ「なし」）
## 未解決・懸念: （なければ「なし」）
```

### Step 3: 実行する

- 依存のないタスクは 1 メッセージ内で複数の Agent 呼び出しを並べ、並列実行する
- 依存のあるタスクは、先行タスクの**監査が通ってから**着手する。報告受領だけでは着手しない
- 実装型サブエージェントが同一ファイル群を触る場合は並列にしない。直列化するか、タスク分割を見直す

### Step 4: 監査する

サブエージェントの報告を受けるたびに、メインセッションで以下のチェックリストを実行する。全項目を通過するまで次のタスクに進まない。

1. **実在確認**: 報告された変更ファイルが実在し、報告どおりの変更が入っているか（Read で該当箇所を確認。全文読みは不要）
2. **完了条件の裏取り**: タスク分解表の完了条件を、機械的事実（コマンド実行結果）で確認する。ビルド・テストが関わる場合は relay-verifier に実行させるか、メインで直接コマンドを叩く。**サブエージェントの「成功しました」報告を裏取りなしに信用しない**
3. **越境確認**: 「対象」外のファイルに変更が漏れていないか（git diff の対象一覧などで確認）
4. **現場判断の裁定**: 報告の「判断したこと」欄を確認し、合意済み設計と矛盾しないか裁定する。矛盾する場合は差し戻す

監査で不合格になったタスクは、不合格理由を明記して同じエージェントに差し戻す。**2 回差し戻しても通らない場合は、そのタスクの進め方自体に問題があるとみなし、作業を止めてユーザーに状況を報告する。**

### Step 5: 停止点で止める

ユーザーの確認やテストが必要なポイントに達したら、そこで止めてユーザーに戻す。どこで止めるかは発動前の設計の詰め具合に依存するが、少なくとも以下では必ず止まる。

- タスク分解表のすべてのタスクが監査を通過したとき（完了報告）
- 合意済み設計の範囲を超える判断が必要になったとき
- 同一タスクが 2 回差し戻しになったとき
- 破壊的操作（削除、履歴改変、外部公開）が必要になったとき

## 難度判定の指針

- 「簡単そう」と「実際に簡単」は異なる。コンテキストの理解や判断を要するタスクは、見た目が単純でも relay-mechanic ではなく relay-implementer-std 以上に割り当てる。純粋に機械的（判断を含まない）と確信できる場合にのみ relay-mechanic を使う
- 迷ったら 1 段上のエージェントに倒す。差し戻しのやり直しコストのほうが、モデル 1 段分の差より高くつく
- effort はエージェント定義に固定されており、スポーン時に上書きできない。調整のレバーは「どのエージェントに割り当てるか」である

## ネストについて

relay-* エージェントは原則として Agent ツールを持たないリーフ（最終実行単位）であり、並列化はメインセッションがファンアウトで束ねる。旧版で許容していた動的ネスト運用は廃止した。1 つのタスク内に「調査してから実装する」のような直列の内部工程がある場合は、タスク分解表で別タスクに切り出し、メインが順に束ねる。

唯一の例外は relay-investigator である。調査は「広く掃いて一つの頭で統合する」形の仕事であり、掃く部分を子に逃がすことで統合用のコンテキストを汚さずに済む。そのため読み取り専用の Explore エージェントへの探索委譲のみを許可している（規律はエージェント定義側に記載）。書き込みを伴う作業のネストは全面的に不可である。

## 自走モード（無人運転）

ユーザーが /goal と併用して発動したとき、または無人での連続実行を明示したときは、以下の規律を追加する。ユーザーの介入は RELAY-STATUS の宣言によって機械的に呼び出すものであり、走行中の成果を随時見てもらうことは想定しない。

### 台帳の永続化

Step 1 のタスク分解表は、会話への出力に加えて TaskCreate でハーネスの Task 台帳に登録する（1 タスク = 1 Task。依存関係は addBlockedBy で表現する）。会話中の表はコンテキスト圧縮で失われうるため、進行状態の正は Task 台帳とする。着手時に in_progress、監査通過時に completed へ更新する。

### 状態宣言

毎ターンの末尾に、次の 1 行を必ず出力する。Step 5 の停止点に達したときは、対応する種別の BLOCKED を宣言して止まる。

`RELAY-STATUS: RUNNING | DONE | BLOCKED(DESIGN | MANUAL-TEST | PERMISSION | STUCK)`

- **RUNNING**: 台帳に未完了タスクが残っており、続行できる
- **DONE**: 全タスクが監査を通過し、完了報告を出した
- **BLOCKED(DESIGN)**: 合意済み設計の範囲を超える判断が必要。再グリル対象の論点を報告に含める
- **BLOCKED(MANUAL-TEST)**: ユーザーの実機確認が必要。確認手順を報告に含める
- **BLOCKED(PERMISSION)**: 権限確認で停止した
- **BLOCKED(STUCK)**: 同一タスクが 2 回差し戻しになった

この行は /goal の評価器が終了判定に使う機械的シグナルである。BLOCKED を宣言すべき局面で RUNNING を続けてはならない。評価器の圧力から逃れるために設計判断を代行することは、停止より悪い失敗である。

### 呼び戻し

DONE / BLOCKED で停止するときは PushNotification で通知する。内容は種別と、ユーザーに求める行動の 1 行とする。

### 開始前チェック

走り出す前に、完了条件の検証に使うコマンド群（テスト・ビルド・lint）が権限プロンプトなしで実行できるかを確認する。未許可のものがあれば開始前にまとめてユーザーに確認する。走行中の権限待ちは、無人運転では停止と同義である。

### 推奨 /goal 条件（定型）

```
/goal RELAY-STATUS: DONE または BLOCKED が宣言されている
```

成果物の正しさは goal 評価器ではなく監査（完了条件の機械的裏取り）が担保する。goal 条件に成果物の状態を書き込む必要はない。DONE は機械的な完了条件の充足を意味し、受け入れ（意図どおりの成果物か）の最終判断は DONE 後のユーザーに残る。

### ループバック

- BLOCKED(DESIGN) → 該当分岐のみ再グリル（ADR / CONTEXT.md 更新）→ 台帳更新 → /goal 再設定で再開
- BLOCKED(MANUAL-TEST) → ユーザーの確認結果を受けて台帳更新 → /goal 再設定で再開
- BLOCKED(STUCK) → タスクの進め方自体を見直し、分解し直すか、ユーザーの指示を仰ぐ

BLOCKED でも goal は達成扱いで自動クリアされるため、介入後の再開は /goal の張り直しを 1 サイクルの単位とする。
